## Model Context Note

The target model is `Qwen/Qwen3-235B-A22B`. Its model card documents a native
32,768-token context and YaRN examples for 131,072 tokens. This plan uses YaRN
factor 8 and `max_position_embeddings=262144` for the requested 256k-token
empirical test, so the final report must state whether 256k actually works on
this stack rather than treating it as an official supported length.

## One-Command Runner

Use this first. It performs the ROCm preflight, runs the configured benchmark,
and writes the Markdown report. If HIP devices are unavailable, it writes a
blocked report instead of attempting to load the 235B model.

```bash
HIP_VISIBLE_DEVICES=0,1,2,3,4,5,6,7 HF_HOME=data/hf_cache \
  .uv-bootstrap/bin/uv run python scripts/run_qwen3_235b_256k_benchmark.py
```

Use `--download-model` on the same command if the model snapshot is not already
present in `data/hf_cache`.

If direct host Python still cannot see HIP devices, use the repo's ROCm Docker
wrapper explicitly:

```bash
HIP_VISIBLE_DEVICES=0,1,2,3,4,5,6,7 HF_HOME=data/hf_cache \
  .uv-bootstrap/bin/uv run python scripts/run_qwen3_235b_256k_benchmark.py \
  --runtime rocm-docker
```

This uses `scripts/rocm_docker_run.sh`, which maps `/dev/kfd` and `/dev/dri`,
adds the render group inside the container, and mounts this workspace read/write.
Because that gives a Docker container GPU device access plus the project files,
run it only when that trust boundary is acceptable.

## Success Gate

The requested target is not absolute compact accuracy alone. The mixed suite includes single-needle, multi-needle, and two-hop retrieval cases. The gate is:

```text
overall_relative_success_to_full >= 0.95
```

If full-cache success is below 0.95, the dataset/run is not a useful proof of
compressor quality and should be rerun with a stronger prompt or adjusted model
settings before optimizing the compressor.
